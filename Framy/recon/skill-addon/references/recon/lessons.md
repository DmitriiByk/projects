# Уроки recon → процедуры (Слой 3)

Каждый урок = воспроизводимый детектор + процедура, а не нарратив. Каждый добавляет проверку в
`dumpSignals()` и/или строку в чек-лист сдачи `build-game-ui`. Формат: Сигнатура промаха → Детектор →
Процедура → Регресс-кейс.

---

## L1 — неполный список lighten-блендов прячет FX-слой

- **Сигнатура промаха.** `has_fx_blend_layer = false` при наличии слоя-энергии/текстуры на бленде
  `LINEAR_DODGE`. Первый список блендов содержал только `LIGHTEN/SCREEN/OVERLAY/SOFT_LIGHT/MULTIPLY`
  и пропускал dodge/burn/plus-семейство. FX-слой (приём T7) не распознавался → при воспроизведении
  потерялась бы текстура.
- **Детектор.** Константа `LIGHTEN_MODES` в `dumpSignals.js` включает `LINEAR_DODGE, COLOR_DODGE,
  PLUS_LIGHTER, PLUS_DARKER, HARD_LIGHT, COLOR_BURN, LINEAR_BURN`. `has_fx_blend_layer` проверяет и
  `node.blendMode`, и `fill.blendMode`.
- **Процедура.** Любой не-NORMAL бленд на слое/заливке = маркер FX-слоя → приём T7 (растр + бленд +
  alpha-маска), держать отдельным переиспользуемым компонентом.
- **Регресс-кейс.** `Button / variant=primary, size=md, state=default` (node `77:35600`), слой
  `Button texture` — IMAGE, opacity 0.15, `LINEAR_DODGE`. Ожидаем `has_fx_blend_layer = true`.

## L2 — текстурный оверлей принят за растровый фон

- **Сигнатура промаха.** `material = "raster"` и `is_raster_bg = true`, хотя картинка — это
  полупрозрачный **оверлей-текстура** (opacity 0.15, lighten-бленд), а база — металлический градиент.
  Слияние «есть IMAGE-fill» → «растровый фон» вело бы к неверной стратегии (9-slice растра T2 вместо
  металл-градиента T8 + текстура T7).
- **Детектор.** Разделить IMAGE-заливки: `textureOverlay` = `opacity < 0.6` ИЛИ lighten-бленд;
  `rasterBase` = `opacity ≥ 0.6` И нормальный бленд И `scaleMode ≠ TILE`. `material` идёт в `raster`
  только при `rasterBase`, иначе `metallic`/`glossy`/`flat`.
- **Процедура.** Сначала классифицировать роль картинки (база vs оверлей), потом выбирать приём.
  Оверлей → T7 поверх; база → T2 (9-slice CROP).
- **Регресс-кейс.** Тот же node `77:35600`: ожидаем `is_raster_bg = false`, `has_texture_overlay =
  true`, `material = "metallic"`.

## L3 — дамп тонкого варианта-обёртки вместо декорированного мастера

- **Сигнатура промаха.** Цель дампа — вариант компонент-сета, который сам почти пустой (`layer_count
  = 2`): обёртка с одним INSTANCE внутри. Реальный декор (фон/notch/арт) лежит глубже — в **мастере**
  этого инстанса (`variant=01`), а не в выбранном узле. Профиль выходит обеднённым → дерево
  недооценивает декор.
- **Детектор.** В `dumpSignals`: если `layer_count` мал И корень содержит один доминирующий INSTANCE,
  чьи дети не раскрылись — пройти в `instance.mainComponent` и до-дампить его поддерево (с пометкой
  `followed_main: <name>`). Эвристика: `reuse_candidates` из одного элемента с `count:1` при низком
  `layer_count` = сигнал «это обёртка».
- **Процедура.** При тонком профиле резолвить и дампить мастер инстанса, а не обёртку. Для карточек —
  целиться в декорированный master-вариант (default/new), там, где живут notch (T9) и арт-слот.
- **Регресс-кейс.** `GameCard` page, вариант `size=sm, ratio=2:3` (node `23705:90697`) → внутри
  `variant=01`. Ожидаем: при follow-into-main `has_ornament_vector`/notch проявляются.

## L4 — 3-слайс не распознан из-за наименования сегментов

- **Сигнатура промаха.** `is_three_slice = false` на кнопках Dragonia, хотя это полноценный 3-слайс:
  сегменты названы `part=left/middle/right` (а у Vegas Hero — `part=left/center/right` + вектор-
  `Polygon` для скоса). Первый детектор искал только суффиксы `=l/=c/=r` или ≥3 `Rectangle` и потому
  пропускал. Это ровно «прошлый раз не справился с Vegas Hero»: угловой 3-слайс не опознавался.
- **Детектор.** `partRe = /part=(left|center|middle|right)|=(l|c|r)$|^(left|center|middle|right)$/i` по
  именам слоёв И по ключам `reuse_candidates`; плюс `polyCount` (вектор-`Polygon`). `sliced =
  partHits>=2 || RectangleCount>=3 || polyCount>=3`. Доп. флаг `is_angled_shape = polyCount>0`.
- **Процедура.** 3-слайс → T1 (углы FIXED, центр FILL). Если `is_angled_shape` — сегменты рисовать
  вектор-полигонами (vector-method), а не прямоугольниками; центр FILL по скосу, углы FIXED.
- **Регресс-кейс.** Dragonia `variant=primary/md/default` (`168691:124812`) → `is_three_slice=true`.
  Vegas Hero `Type=primary/md/default` (`168693:127621`) → `is_three_slice=true`, `is_angled_shape=true`.

## L5 — не доверяй одному скриншоту: цвет базы бери с канонического мастера

- **Сигнатура промаха.** По одному скрину варианта из сета (`Type=primary` `168693:127621`) я решил,
  что корпус тёмный, и пересобрал кнопку тёмной. На деле канонический фон `CtaPrimaryBg/State=default`
  (`168693:127542`) — **зелёный** (`#6FEA5C`): база зелёная, тёмный лишь виньет 16%. Тёмный вид того
  варианта — это инстанс-override поверх зелёного мастера, а не базовый цвет. Я переусердствовал в
  «исправлении».
- **Детектор.** Цвет базы определять по **мастеру** декор-фона (самая крупная непрозрачная заливка
  корпуса), а не по одному рендеру инстанса. Если инстанс выглядит иначе — это override; снять оба и
  сравнить (`dominant_fill` на мастере + `instance_overrides`).
- **Процедура.** Брать доминирующую заливку с мастера; инстанс-вариации (тёмный/светлый) применять как
  override отдельным слоем, не путать с базой. Лейбл-цвет — по контрасту к РЕАЛЬНОЙ базе варианта.
- **Регресс-кейс.** `CtaPrimaryBg/State=default` (`168693:127542`) → база зелёная `#6FEA5C`; вариант
  сета `168693:127621` тёмный = override. Реконструкция v2 (`169092:2`) собрана по зелёному мастеру.

## L6 — свечение даётся СТЕКОМ полигонов с падающей opacity, не градиентом

- **Сигнатура промаха.** Краевое зелёное свечение Vegas Hero я воспроизвёл линейным градиентом — вышло
  плоско и неверно. На деле каждый крайний слайс = **3 наложенных полигона одного цвета** разной формы
  и размера: тонкий треугольник `opacity 1` сверху + два пентагона `opacity 0.3` под ним. Объём и
  мягкий фолл-офф рождаются из ПЕРЕКРЫТИЯ полупрозрачных слоёв (чем глубже слой — тем меньше opacity).
  Это прямое применение принципа build-game-ui «глубина = СТЕК слоёв, не один fill» — я его знал, но в
  сборке нарушил.
- **Структура (факты).** Слайсы: center тянется, left/right зеркальны. Крайний слайс `Property 1=left`
  (20×42), полигоны повёрнуты −90°: `Polygon 1` (треуг., 12×38, op1, сверху) · `Polygon 2` (15×42,
  op0.3) · `Polygon 3` (20×36, op0.3, глубже всех). Все `#6FEA5C`.
- **Детектор.** Флаг `has_layered_opacity_stack`: ≥2 соседних узла одного цвета с разной `opacity`
  (особенно ступенька 1.0 → 0.3) и схожей формой → это glow-стек, воспроизводить полигонами, НЕ
  градиентом. Сохранять per-layer `opacity` (уже есть в дампе — надо использовать при сборке).
- **Процедура.** Узнал glow-стек → реплицировать N полигонов с теми же opacity и порядком (back→front:
  крупные/тусклые → мелкий/яркий). Цвет один, форма вложенная. Зеркалить для противоположного слайса.
- **Регресс-кейс.** Vegas Hero `Property 1=left` (`168693:127525`) → 3 полигона, opacity [0.3,0.3,1],
  зелёные; реконструкция `cap-test-left` (`169089:2`) совпала по виду.

## L7 — авторинг «по памяти» вместо реконструкции из данных (корневой)

- **Сигнатура промаха.** Ребилды v1–v2 промахивались системно (форма капл, пропорции, база цвета,
  чёрный контур, плоское свечение), потому что я АВТОРИЛ геометрию приблизительно, а не пересоздавал из
  реальных значений узла. Это родитель почти всех остальных уроков.
- **Детектор/инструмент.** `recon/faithful-rebuild.js` → `recreate(src, parent)`: пересоздаёт каждый
  узел из его реальных `vectorPaths` / `relativeTransform` / `fills` / `opacity` / `effects`, гасит
  дефолтный чёрный stroke, делает фолбэк шрифта. Геометрия/цвета/пропорции физически не могут быть
  «на глаз».
- **Процедура.** Дефолт ребилда: `dumpSignals` → `mapStrategy` (понимание/отзывчивость) →
  `recreate` (точная сборка из данных) → скриншот-сверка по `build-checklist.md`.
- **Регресс-кейс.** Vegas Hero `CtaPrimaryBg/State=default` (`168693:127542`): v3/v4 через `recreate`
  совпали с оригиналом (зелёный шестиугольник, послойное свечение, текстура, тёмный лейбл) — там, где
  ручные v1/v2 промахивались.

## L8 — шрифт грузится без проверки доступности → молчаливый провал

- **Сигнатура промаха.** Лейбл не прорисовался: оригинал на `Bebas Neue Pro Bold Italic`, которого нет
  в окружении (есть только `Bebas Neue Regular`, `Jost`). `loadFontAsync` бросал, рекурсия глотала
  ошибку — текст пропал.
- **Детектор.** Перед загрузкой — `listAvailableFontsAsync`; если точного нет → тот же family любой
  стиль → family по префиксу (`Bebas Neue Pro`→`Bebas Neue`) → `Inter`. Реализовано в
  `_loadFontWithFallback` (`faithful-rebuild.js`).
- **Процедура.** Никогда не грузить шрифт «вслепую»; всегда фолбэк, и зафиксировать какой использован.
- **Регресс-кейс.** Лейбл `BUTTON` (`#121723`) собран на `Bebas Neue` как замена `Bebas Neue Pro`.

## L9 — GROUP→FRAME оставляет дефолтную белую заливку

- **Сигнатура промаха.** При реконструкции исходные GROUP я пересоздавал как FRAME. У GROUP нет
  свойства `fills`, условие `if(Array.isArray(s.fills))` пропускало присвоение → созданный FRAME
  оставался с дефолтной БЕЛОЙ заливкой. Итог: белые боксы за прозрачными каплями + «лишние элементы
  справа» (белые углы вокруг шеврона).
- **Детектор/правило.** Всегда задавать `fills` явно: `n.fills = Array.isArray(s.fills) ?
  clone(s.fills) : []`. Любой созданный контейнер без явной заливки = подозрение на дефолтный белый.
- **Процедура.** После сборки — скан на видимые белые SOLID-заливки у контейнеров, которых не было в
  оригинале; обнулять. (Зашито в `faithful-rebuild.js`.)
- **Регресс-кейс.** v4: `Group 1410104127`/`1410104128` имели белый фон → обнулены; кнопка чистая.

## L10 — повёрнутая/зеркальная GROUP: ставить листья по absoluteTransform, не по матрице групп

- **Сигнатура промаха.** Правый кап Vegas Hero зеркалится через `Group` rot 180°. Две попытки
  провалились: (v4) пересоздание GROUP→FRAME — разные системы координат, два полигона не зеркальны
  («X» справа); (v5) ручное перемножение `relativeTransform` группы с детскими — почти верно, но один
  полигон уезжал на ~6.79px (дрейф трансляции при композиции поворота).
- **Детектор/правило.** Не воссоздавать иерархию групп вручную. Ставить **каждый ЛИСТ** по его
  `absoluteTransform`, приведённому к новому корню: `n.relativeTransform = mul(inv(root.absoluteTransform),
  leaf.absoluteTransform)`. `absoluteTransform` уже запекает все повороты/зеркала/масштабы предков —
  ни GROUP→FRAME рассинхрона, ни дрейфа. Реализовано в `faithful-rebuild.js → rebuildFaithful()`.
- **Цена/процедура.** Иерархия схлопывается (листья под корнем) — для точной визуальной реконструкции
  это ок; 3-слайс-отзывчивость, если нужна, строится отдельно по `mapStrategy`. После сборки —
  **числовая** сверка зеркальности лево/право (не только скриншот: скрин дважды меня обманул).
- **Регресс-кейс.** Vegas Hero правый кап (`Group 1410104128`, rot 180): v4/v5 ломались; v6 через
  `absoluteTransform` → `allMirrored: true` (полигоны 0.3/0.3/1 точно зеркальны).

## L11 — ребилд обязан быть на компонентах и тянуться (не плоская копия)

- **Сигнатура промаха.** Плоская реконструкция (`rebuildFaithful`) визуально точна, но структурно
  неверна: фон — набор листьев, НЕ компоненты, и кнопка НЕ тянется. Я проигнорировал `reuse_candidates`,
  которые `dumpSignals` нашёл ещё в начале (части `part=left/center/right`, `Button texture`), и собрал
  имитацию вместо реюза — прямое нарушение доктрины build-game-ui «реюз > имитация» и Слоя 0.
- **Детектор/правило.** Если в профиле есть `reuse_candidates` (существующие компоненты/части) —
  собирать кнопку **из их инстансов** в auto-layout 3-слайсе (каплы FIXED, центр FILL), а не
  реконструировать геометрию. Фон = инстанс компонента (напр. `CtaPrimaryBg`), не плоские векторы.
  Плоский `rebuildFaithful` — только для пиксельной сверки/когда компонентов нет.
- **Процедура.** dump → mapStrategy → **если reuse_candidates есть: assemble из инстансов (3-слайс,
  центр FILL, тянется)**; иначе flat-реконструкция. Проверить растяжение на 2 ширинах.
- **Регресс-кейс.** Vegas Hero primary: финальная кнопка = инстанс `CtaPrimaryBg` (`168693:127542`) +
  лейбл; ширина 148→280 тянется корректно (центр растёт, каплы на месте, шевроны целы).

## Кандидаты в чек-лист сдачи build-game-ui (из L1–L11)
- [ ] Каждый не-NORMAL бленд распознан как FX-слой и вынесен в компонент (не потерян)?
- [ ] Роль каждой картинки классифицирована (база/оверлей-текстура) до выбора приёма?
- [ ] Цель дампа — декорированный мастер, а не тонкий вариант-обёртка (провалиться в mainComponent)?
- [ ] 3-слайс опознан по part-наименованию/polygon-сегментам, а не только по числу Rectangle?
- [ ] Определена доминирующая база vs краевые акценты (не перепутаны); лейбл-цвет инверсен базе?
- [ ] Свечение/объём собрано стеком полупрозрачных полигонов (по per-layer opacity), а не градиентом?
- [ ] Ребилд сделан из данных (`recreate`), а не «по памяти»; пройден `build-checklist.md` со скриншот-сверкой?
- [ ] Шрифт проверен на доступность до загрузки, иначе фолбэк (не молчаливый провал)?
