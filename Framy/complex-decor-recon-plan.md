# План: работа со сложным декором (recon → reconstruction)

**Цель.** Сделать переносимым (не под одну кнопку) навык: *распознать* форму сложного декоративного
элемента и *воспроизвести* её с отзывчивостью. Корень — то, что отрабатывалось через «трюки» и
разбор кнопок в `build-game-ui`.

**Что уже есть (фундамент, не переписываем).**
- `build-game-ui` — «железное правило» (бери факты из файла, не угадывай), порядок разведки
  (`get_metadata → get_design_context → get_variable_defs → дамп слоёв`), каталог приёмов **T1–T15**,
  переносимые принципы, чек-лист сдачи, грабли `use_figma`.
- `decoration-craft` — лестница «лёгкий→тяжёлый приём» (рунги 1–11), таблица toolkit, responsive-дисциплина.
- `doctrine` — Entry #1 (негативный gap), Variation A (auto-layout padding gap), Rule 10 (per-variant
  matrix), Background Implementation Verification.

**Три пробела (из аудита), которые закрываем.**
1. **Разведка делается «на глаз».** Шаг 2 build-game-ui перечисляет *что снять*, но нет одного
   исполняемого скрипта, который выдаёт стабильный JSON-профиль. → **Слой 1: `dumpSignals()`**.
2. **Нет моста профиль→приём.** Дерево T1–T15 выбирается человеком по визуальным признакам;
   нет детерминированного маппинга «сигнал в профиле → приём». → **Слой 2: дерево решений**.
3. **Урок-провал записан как разбор, а не процедура.** Нельзя повторить. → **Слой 3: урок-процедура**.

## Слой 0 — режим работы: реюз vs повтор своими руками

Сквозное правило, выбирается ДО разведки и управляет всеми слоями.

- **Режим «копирование/прод» (по умолчанию).** Если нужный внутренний компонент (Bg / Decor-рама /
  FX-слой / лейбл) уже существует и совпадает по профилю — **переиспользуем инстансом**, не
  перерисовываем геометрию. Это прямое следствие принципа build-game-ui «Реюз > имитация» и чек-листа
  (один `imageHash` / одна геометрия на все состояния). Перекрас под тему = клон градиента + замена
  стопов, а не дубль.
- **Спросить оператора при неоднозначности.** Если найден кандидат на реюз, но совпадение неполное
  (другой размер/материал/частичное пересечение признаков) — **спросить**: (a) переиспользовать
  существующий, (b) клонировать и доработать, (c) собрать заново. Не угадывать молча.
- **Режим «обучение новым трюкам».** Цель — натренировать приём, поэтому **сознательно НЕ
  переиспользуем**, а пытаемся воспроизвести своими руками (вектор/слои/эффекты), даже если готовый
  компонент есть. Реюз тут обнулил бы смысл упражнения.

**Как режим влияет на слои:** Слой 1 (`dumpSignals`) дополнительно ищет существующие компоненты-кандидаты
на реюз (по совпадению профиля) и кладёт их в `reuse_candidates[]`. Слой 2 в режиме «копирование»
сначала проверяет `reuse_candidates[]` и только при промахе идёт в дерево приёмов; в режиме «обучение»
дерево вызывается всегда. Дефолт режима задаёт оператор; при сомнении — спросить.

## Слой 1 — `dumpSignals()`: исполняемый recon-дамп

Один скрипт под `use_figma`, который по узлу-референсу обходит дерево и возвращает **стабильный
JSON-профиль**. Заменяет ручной осмотр слоёв детерминированным снимком. Живёт в
`references/figma-recipes.md` секции «Дамп» (расширяем существующую), вызывается первым в Шаге 2.

**Снимаемые сигналы (по каждому узлу дерева):**
- геометрия: `type`, `width/height`, `cornerRadius` (и per-corner), `vectorPaths` (есть/нет, windingRule),
  `layoutMode`/`primaryAxisSizingMode`/`counterAxisSizingMode`, `constraints`, `layoutGrow`/`layoutAlign`;
- заливки: `fills[]` — тип (SOLID/GRADIENT_LINEAR/RADIAL/ANGULAR/IMAGE), стопы, `gradientTransform`,
  `scaleMode` (для IMAGE: FILL/FIT/CROP/TILE) + наличие `imageHash`;
- кромки: `strokes[]`, `strokeWeight`, `strokeAlign`, gradient-stroke (да/нет);
- глубина: `effects[]` — DROP_SHADOW / INNER_SHADOW / LAYER_BLUR / BACKGROUND_BLUR + параметры;
- смешивание: `opacity`, `blendMode` (LIGHTEN/SCREEN/MULTIPLY/OVERLAY/SOFT_LIGHT — маркеры FX-слоёв);
- маски: `isMask`, наличие boolean/evenodd (негативное пространство);
- структура: число детей, глубина, повторяющиеся имена (FX/Bg/Decor как отдельные компоненты),
  9-slice-признаки (FIXED-углы + FILL-центр).

**Производные флаги профиля (вычисляются из сигналов — это и есть вход дерева):**
`stretches_horizontally`, `is_raster_bg`, `has_inner_shadow`, `has_lift_shadow`, `has_edge_glow`,
`has_gradient_sheen`, `has_ornament_vector`, `has_notch`, `has_fx_blend_layer`, `is_pill`,
`is_inset` (вдавленный, для инпутов), `texture_shape` (есть направленная текстура / нет), `material`
(flat / glossy / metallic / neon-glass), `states[]`.

**Формат выхода (контракт).** Один JSON: `{ root, layers[], flags, states, reuse_candidates[], proposals[] }`,
где `reuse_candidates[]` — существующие компоненты, совпавшие по профилю (вход для Слоя 0/2), а
`proposals[]` — кандидаты в новые токены (по правилу doctrine — раз концепт-цвета не баним в config).
Стабильность важнее полноты: порядок ключей и имён фиксирован, чтобы профили двух элементов
сравнивались диффом.

**Фолбэк резолва узла (известный блокер).** `getNodeByIdAsync` флапает транзиентным null. Скрипт:
1) пробует `getNodeByIdAsync(id)`; 2) при null — `figma.currentPage.findOne(n => n.id === id)`;
3) ретрай N раз с паузой; 4) если узел на другой странице — переключить `currentPage` на нужную
страницу (Button) и повторить. Логировать, какая ветка сработала.

## Слой 2 — дерево «профиль → стратегия»

Детерминированная таблица: набор `flags` из Слоя 1 → приём(ы) `build-game-ui` T1–T15 + рунг
`decoration-craft`. Не заменяет визуальное дерево T1–T15, а делает его **вызываемым из профиля**.

| Сигнал в профиле | Приём | Источник |
|---|---|---|
| `stretches_horizontally` + прямые/скошенные края | 3-слайс (L FIXED / C FILL / R FIXED) | T1 |
| `is_raster_bg` + тянется | 9-slice растра `scaleMode:CROP` + общие доли швов | T2 |
| `material:neon-glass`, объём у кромок | alpha-градиенты кромок + 30% wash + текстура | T3 |
| `has_gradient_sheen` + гладкий глянец | вертикальный градиент + блик + фаска + INNER_SHADOW | T4 |
| `is_pill` + мягкий объём | жёсткая тень blur0 снизу + тонкий белый внутр. stroke | T5 |
| светящееся ядро (radial + inner) | радиальный градиент + INNER_SHADOW | T6 |
| `has_fx_blend_layer` (LIGHTEN/энергия) | РАСТР + LIGHTEN + alpha-маска; анимация → T13 | T7 |
| `material:metallic`, рама-оправа | концентрические rounded-rect с градиентами | T8 |
| `has_ornament_vector` / `has_notch` | отдельный вектор-компонент, зеркало | T9 / decoration-craft «scalable-frame» |
| блестящая поверхность | верхний глянец + диагональ + INNER_SHADOW | T10 |
| `has_lift_shadow` (выдавлено из фона) | внешняя DROP_SHADOW мягкая+плотная + нижний блик | T11 |
| `is_inset` (инпут, вдавлено) | инверсия T11: inner shadow тёмный сверху-внутрь | T11-inv |
| лейбл должен «играть» | градиентная заливка текста + жёсткая тень 0/1 | T12 |
| `states[]` непустой | один скелет, меняем свет/насыщ./glow/FX; компонент-сет | T14 |
| анимация только hover | GIF-фон + контент слоем + прототип ON_HOVER | T13 |
| `texture_shape` направленная | фикс. сетка `constraints:MIN`, база тянется отдельно | T15 |

**Правило приоритета (из decoration-craft).** При нескольких совпадениях — брать **самый лёгкий
рунг**, читающийся так же на целевом размере. Не прыгать на image/chrome, если gradient+shadow
неотличимы. Глубина = СТЕК слоёв, не один fill.

**Связка с tier (doctrine Rule 0).** Богатый декор оправдан на 🟩 High-Custom; на Standard/🟧
Low-Custom — token fill+stroke+radius и стоп. Дерево всегда проверяет tier перед выбором тяжёлого приёма.

## Слой 3 — урок-провал → процедура

Превратить разбор провалённой орнаментальной кнопки в **воспроизводимый чеклист**, а не в нарратив
«что пошло не так». Формат каждого урока:

- **Сигнатура промаха** — какой `flag` профиля был пропущен/неверно прочитан (напр. `has_lift_shadow`
  не снят → потеряна внешняя тень-отрыв; `texture_shape` прочитан как «горошек»).
- **Детектор** — конкретная проверка на JSON-профиле, ловящая этот класс (войдёт в чек-лист сдачи).
- **Процедура исправления** — шаги приёма из Слоя 2, которые надо было применить.
- **Регресс-кейс** — ссылка на узел-эталон, на котором проверяется, что промах больше не повторяется.

Уроки копятся в `references/lessons.md` (новый файл скилла) и каждый добавляет строку в чек-лист сдачи
build-game-ui. Так провал один раз → защита навсегда.

## Фазы исполнения

1. **P1 — `dumpSignals()` рабочий прототип.** Дописать резолв-фолбэк, прогнать на той орнаментальной
   кнопке, получить валидный JSON-профиль. *Готово, когда:* профиль снимается без ручного осмотра и
   повторяется (идемпотентность дампа).
2. **P2 — дерево как код.** Маппинг-таблица + функция `flags → [приёмы]`. *Готово, когда:* на 3
   разных элементах (кнопка, карточка с notch, инпут-inset) дерево выдаёт верный набор приёмов.
3. **P3 — урок-процедура.** Перенести провал кнопки в `lessons.md` по формату Слоя 3 + строки в чек-лист.
   *Готово, когда:* `dumpSignals` ловит ту же ошибку детектором.
4. **P4 — обобщение.** Прогнать петлю recon→дерево→сборка на не-кнопке (карточка/панель), убедиться,
   что приёмы переносятся. *Готово, когда:* воспроизведение нового типа без правки дерева.

## Критерии успеха (весь план)
- Разведка любого декор-элемента = один вызов, стабильный JSON (не ручной осмотр).
- Выбор приёма выводится из профиля, а не из памяти.
- Каждый прошлый промах имеет детектор в чек-листе.
- Петля работает на ≥2 типах элементов без переписывания.
