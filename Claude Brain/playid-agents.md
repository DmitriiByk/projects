# PlayID — кастомные агенты (Flutter/Riverpod)

4 скилла, созданные для мобильного трека PlayID (store-аппки + прототипы на телефоне). Стек по умолчанию: Flutter + Riverpod. Все на русском.

## Агенты
- **ux-ui-researcher** — стеко-нейтральный. Ищет UX/UI-паттерны и референсы, даёт 2–3 варианта с плюсами/минусами + рекомендацию, напоминает про краевые состояния и доступность. Применим к ЛЮБОМУ проекту (и веб, и мобайл).
- **flutter-feature-architect** — проектирует фичу/экран до вёрстки: feature-first структура папок, состояния через AsyncValue, провайдеры, поток данных, скелет кода. Riverpod заложен как стандарт (можно переключить на BLoC).
- **design-system-keeper** — двусторонний синк Figma ↔ W3C JSON токены + аудит консистентности + раскладка в Flutter ThemeData.
- **image-asset-handler** — подготовка ассетов: WebP/сжатие, нарезка @1x/@2x/@3x под Flutter, отчёт «было → стало».

## Статус
- Установлены как user-скиллы (namespace anthropic-skills:*). Срабатывают по контексту автоматически или явно через /имя.
- image-asset-handler на момент создания мог быть не сохранён (надо нажать Save skill).
- Написаны "на чутьё", не прогнаны через evals. При желании — отшлифовать через skill-creator.

## ВАЖНО: граница с Forge-X
- Эти агенты — для PlayID (мобайл), НЕ для Forge-X (веб). flutter-feature-architect к React/Tailwind-коду Forge-X не применять.
- **Конфликт-риск:** design-system-keeper делает Figma↔JSON↔код двусторонне. Если натравить на ForgeX Library — НЕ запекать concept-значения и НЕ менять топологию коллекций (нарушит доктрину Forge-X / "Path B"). Он должен только ЧИТАТЬ дизайн из Figma и раскладывать в ThemeData PlayID, а не переписывать источник.
